### **Название задачи:**

Интеграция с платформой "Умный дом"

### **Автор:**

Архитектурная команда PropDevelopment

### **Дата:**

25.03.2026

### **Контекст и описание проблемы**

Бизнес поставил задачу расширить функциональность для собственников жилья, добавив сервисы "Умный дом": интеллектуальный домофон и шлагбаум. Эти сервисы предоставляет внешний технологический партнёр.

Необходимо спроектировать безопасную, масштабируемую и поддерживаемую архитектуру для интеграции с платформой партнёра. Прямая интеграция в существующий монолитный сервис `tenant-core-app` приведёт к его усложнению, сильной связанности (tight coupling) с API конкретного партнёра и нарушению принципа единственной ответственности (Single Responsibility Principle). Требуется решение, которое изолирует логику интеграции и обеспечивает высокий уровень безопасности, так как система будет управлять физическим доступом к жилым комплексам.

### **Решение**

Принято решение реализовать интеграцию через новый выделенный микросервис, который будет выполнять роль шлюза интеграции (Integration Gateway). Этот подход позволит инкапсулировать всю логику взаимодействия с партнёром, обеспечив слабую связанность с остальной системой PropDevelopment.


#### 1\. Организация взаимодействия

Взаимодействие между системами разделяется на два потока: синхронный (команды от пользователя) и асинхронный (события от устройств).

* Синхронный поток (команды):

  1. Собственник инициирует действие в Мобильном приложении (например, «Открыть шлагбаум»).
  2. Запрос поступает в tenant-core-app, который проверяет права пользователя.
  3. tenant-core-app делегирует команду новому сервису smart-home-service через внутренний REST API.
  4. smart-home-service отправляет защищённый API-запрос на платформу партнёра.
  5. Платформа партнёра выполняет команду и возвращает синхронный ответ о результате, который транслируется обратно пользователю.
* Асинхронный поток (события):

  1. На устройстве происходит событие (например, звонок в домофон).
  2. Платформа партнёра отправляет подписанный Webhook на эндпоинт в smart-home-service.
  3. smart-home-service валидирует подпись, обрабатывает событие (например, находит собственника по ID устройства) и инициирует отправку push-уведомления.

#### 2\. Протоколы аутентификации и авторизации

* Аутентификация (сервер-сервер): для взаимодействия между smart-home-service и API партнёра используется протокол OAuth 2.0 с типом предоставления Client Credentials Grant. Наш сервис использует Client ID и Client Secret для получения временного Bearer-токена, который включается в каждый API-запрос.
* Авторизация: проверка прав конечного пользователя (собственника) на управление конкретным устройством остаётся в зоне ответственности tenant-core-app. smart-home-service оперирует уже авторизованными действиями, проверяя только права на уровне системы через области (scopes) OAuth-токена.

#### 3\. Требования к безопасности

|Требование|Описание|
|-|-|
|Шифрование канала|Всё взаимодействие между серверами PropDevelopment и платформой партнёра должно осуществляться исключительно по HTTPS с использованием TLS 1.2+.|
|Защита API|API партнёра должно быть защищено от OWASP API Security Top 10 и включать механизмы Rate Limiting для защиты от DoS-атак.|
|Аутентификация Webhooks|Все входящие вебхуки должны быть подписаны с использованием HMAC-подписи (например, в заголовке X-Hub-Signature) для защиты от поддельных запросов. smart-home-service обязан проверять эту подпись.|
|Принцип минимальных привилегий|PropDevelopment передаёт партнёру только минимально необходимый объём данных (например, анонимные идентификаторы). Биометрические данные и полные списки номеров автомобилей хранятся и обрабатываются только на стороне партнёра.|
|Аудит и логирование|Платформа партнёра должна вести и предоставлять по запросу журналы всех API-вызовов, связанных с изменением прав доступа и командами на открытие устройств.|

### **Альтернативы**

1. Интеграция логики напрямую в tenant-core-app
Причина отказа: этот подход создаёт сильную связанность (tight coupling) с API конкретного партнёра, что усложнит переход на другого поставщика в будущем. Он также нарушает принцип единственной ответственности, перегружая tenant-core-app дополнительной, непрофильной логикой и увеличивая риски безопасности для основного сервиса.
2. Использование готового iPaaS-решения (Integration Platform as a Service)
Причина отказа: для интеграции с одним партнёром внедрение полнофункциональной iPaaS-платформы является избыточным и дорогостоящим. Разработка собственного легковесного микросервиса-шлюза даёт больше гибкости и является более экономически целесообразным решением в данном контексте.

**Недостатки, ограничения, риски**

|Риск|Митигация|
|-|-|
|Появляется ещё один сервис (smart-home-service) и БД, которые нужно разворачивать, мониторить и поддерживать|Использование унифицированных CI/CD-пайплайнов, стандартных дашбордов для мониторинга и логирования|
|Возможные сбои или высокая задержка на стороне партнёра могут повлиять на доступность сервиса|В `smart-home-service` должны быть реализованы паттерны устойчивости: Circuit Breaker, Retries и Timeouts|
|Если smart-home-service будет недоступен, входящие события могут быть утеряны|Использовать асинхронную обработку, помещая входящие вебхуки в отказоустойчивую очередь (например, RabbitMQ или Apache Kafka) перед обработкой. Согласовать с партнёром механизм повторной отправки вебхуков в случае сбоя|
|Компрометация Client Secret или ключа для подписи. Утечка этих данных позволит злоумышленнику отправлять команды от имени PropDevelopment|Хранить секреты в защищённом хранилище (например, HashiCorp Vault) и использовать механизм регулярной ротации ключей|
