# Отчёт по результатам анализа Kubernetes Audit Log

## Подозрительные события

1. Доступ к секретам:

   * Кто: system:serviceaccount:secure-ops:monitoring
   * Где: kube-system
   * Почему подозрительно: сервисный аккаунт monitoring пытается получить доступ к секретам в namespace kube-system. Это может быть попыткой кражи токенов системных компонентов или другой чувствительной информации.
2. Привилегированные поды:

   * Кто: minikube-user
   * Комментарий: Привилегированные поды имеют доступ к узлу хоста и могут использоваться для эскалации привилегий и компрометации кластера.
3. Использование kubectl exec в чужом поде:

   * Кто: minikube-user
   * Что делал: просматривал содержимое файла /etc/resolv.conf в поде coredns.
4. Создание RoleBinding с правами cluster-admin:

   * Кто: minikube-user
   * К чему привело: сервисный аккаунт monitoring получил полный контроль над кластером.
5. Удаление audit-policy.yaml:

   * Кто: admin
   * Возможные последствия: Отключение аудита позволяет скрыть следы злонамеренной активности. Это одна из первых целей при компрометации кластера.

## Вывод

Анализ аудит-логов выявил несколько критических инцидентов, указывающих на попытку компрометации кластера. Основные проблемы связаны с недостаточной конфигурацией RBAC, позволяющей создавать привилегированные поды и эскалировать привилегии через RoleBinding.
